bonfireDB — Co-founder Plan

The open-source, agent-native clinical backend. Your health agent reads a cited, governed, token-lean view instead of raw FHIR, so it is measurably cheaper and more accurate, and you wrote zero FHIR. Updated 2026-06-25 — research-backed rewrite, greenfield, 14-day MVP.

In one line. Agents are bad at clinical data (they cap near ~50% accuracy on raw FHIR) and clinical agent loops are expensive (a full FHIR record runs to millions of tokens). bonfireDB fixes both with one design decision: the agent never reads raw FHIR, it reads a Cited Context Projection we build from canonical FHIR underneath. We prove it with an open benchmark. We win as the nice way to build AI health apps: start with AI scribes, become the default backend.

Thesis & why now

  • The problem is acute, not a convenience. Two independent benchmarks cap LLM agents at ~42-50% on raw FHIR (FHIR-AgentBench; FHIRPath-QA). Handing an agent raw FHIR query power actually lowered accuracy. Input format alone swings accuracy ~16 points (Markdown-KV 60.7% vs JSON 52.3% vs CSV 44.3%). Naive FHIR-into-context hallucinates like raw text ("Representation Before Retrieval", medRxiv 2026).
  • Cheaper and more accurate is the same decision. A clean, cited, typed slice both costs fewer tokens and scores higher (CLEAR, npj Digital Medicine 2024: F1 0.90 vs 0.79 at ~70-82% fewer tokens). We do not trade cost against quality; we improve both with one move.
  • Why greenfield, not migration. Supabase, Firebase, Convex, and Medplum all won on greenfield. Migrating a live team to an unproven backend is a tarpit. Narrow the beachhead (scribes); keep the vision big.
  • Why a backend, not an app. The AI-scribe end market is consolidating (Abridge, Microsoft/Nuance, Epic native). We do not compete with them; we are the governed data layer the next thousand AI-health apps build on.

Architecture in one picture

One idea. bonfireDB is a governed read-and-act path the agent is forced through. We control the projection (accuracy), the gate (safety), and the audit (proof). That integrated depth is what a competitor cannot clone in a week. 
            WRITE (one atomic tx)                         READ (agent / app / SDK)
  typed primitive --map--> FHIR R4 -----+              +--> typed projections (vd_*)  <5ms
                                        |              |
   INSERT fhir_resources (JSONB canon) -+--> COMMIT ---+--> Cited Context Projection (compact, span-cited)
   INSERT history + write_inputs        |              |
   UPSERT typed projection (same tx)    |              +--> FHIR search (spidx) / 1-call FHIR export
   INSERT outbox --(async)--> embeddings, reindex      |
                                                       +--> typed MCP tools (propose-only, governed)

  every row: practice_id + RLS fail-closed   |   every read: ABAC policy receipt + hash-chained audit
  • FHIR is the source of truth; the primitive is just the write API. Keep the raw typed payload (write_inputs) so we can re-derive FHIR when the mapping improves. One write path, never a dual write.
  • Read off indexed projections, not JSONB scans (the Medplum/HAPI model; Aidbox-style JSONB-direct search does not scale). The read side is droppable and rebuildable from canonical FHIR.
  • The agent reads the Cited Context Projection, never a 50KB FHIR bundle. Raw FHIR is an explicit, rarely-needed escape hatch.

The proof: BTAB (Bonfire Token + Accuracy Bench)

The benchmark is the marketing artifact. The audience fact-checks; intellectual honesty is the GTM moat.

  • Three arms of identical content: raw FHIR R4 bundle (the foil), compact FHIR JSON, and the Bonfire Cited Context Projection.
  • Named tokenizer per model (state model + tokenizer or it is not reproducible). Token delta runs offline with zero keys; the accuracy arm needs the reader's own model keys and runs on main — we say so plainly.
  • Honest metrics: median tokens/query, task accuracy on a fixed golden-query suite, citation precision/recall, and the primary metric accuracy per 1K tokens.
  • Anti-contamination: SHA-256 pre-registration of held-out splits; we cite the benchmark that disagrees with us; we never quote someone else's headline multiplier as ours.
  • v0 scope: synthetic corpus only (the MIMIC benchmark is post-launch and never goes through a cloud sandbox).

Compliance (honest)

  • Runs in your cloud, under your BAA — no vendor BAA to negotiate.
  • HIPAA-ready, in code — ships the 45 CFR 164.312 safeguards: RLS → access control, hash-chained log → audit + integrity, identity → authentication, TLS → transmission. Designed to the January 2025 HIPAA NPRM floor (AES-256, TLS 1.3, MFA-ready, immutable audit).
  • FDA posture — cited context + propose-only + a clinician who can independently review the basis maps onto the 2026 Non-Device CDS safe harbor. We warn builders that risk scores can cross into device territory.
  • You own compliance; we remove the grunt work — never "compliant out of the box." SOC 2 + a fully-managed BAA mode are the enterprise tier.

Risks & mitigations

1. The accuracy/token claim is unproven until measured. Build BTAB early (days 9-11) and publish Bonfire's own number with a stated baseline + tokenizer. If we cannot beat the baseline, fix the projection before marketing it.
2. The primitive-to-FHIR mapping is the hard 80%. Wrong FHIR is a compliance failure, and "lossless" is a moving target (US Core + terminology churn). The scribe beachhead keeps it to ~8 resources; the loss-ledger + golden round-trip fixtures keep it honest.
3. A cross-tenant leak is one dead company. RLS fail-closed on every row from row zero (SQL, vector, agent), plus ABAC policy receipts. Retrofitting security is how you leak — so we build it first, not last.
4. Market & substitutes. "Supabase + my own schema" is the path of least resistance until interop is forced — counter with "you will regret skipping FHIR at month 12" and a quantified rewrite cost. The greenfield-scribe market is consolidating — so we sell the backend the next wave of agents needs, and add the import path so we are useful to existing-FHIR teams too.
